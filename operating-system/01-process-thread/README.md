# 실습 1: 프로세스와 스레드

## 면접 키워드
- PCB (Process Control Block)
- 프로세스 상태 전이 (5-State Model)
- Context Switch
- 프로세스 vs 스레드

## 과제
프로세스의 상태 전이를 시뮬레이션하고, Context Switch 과정을 직접 구현합니다.
스레드와 프로세스의 메모리 공유 차이를 실험으로 체감합니다.

## 면접 예상 질문

### Q. 프로세스와 스레드의 차이를 설명하세요.

**프로세스**는 운영체제로부터 자원을 할당받는 **작업의 단위**이고, **스레드**는 프로세스 내에서 실행되는 **실행의 단위**입니다.

| 구분 | 프로세스 | 스레드 |
|------|---------|--------|
| 메모리 | 독립된 주소 공간 (Code, Data, Heap, Stack) | Code, Data, Heap **공유**, Stack만 독립 |
| 생성 비용 | 큼 (메모리 복사, PCB 생성) | 작음 (스택만 할당) |
| Context Switch | 느림 (TLB flush, 캐시 무효화) | 빠름 (같은 주소 공간) |
| 통신 | IPC 필요 (Pipe, Socket 등) | 공유 메모리로 직접 접근 |
| 안정성 | 한 프로세스 crash → 다른 프로세스 영향 없음 | 한 스레드 crash → 전체 프로세스 종료 |

```
프로세스 A              프로세스 B
┌──────────────┐     ┌──────────────┐
│ Code         │     │ Code         │
│ Data         │     │ Data         │
│ Heap         │     │ Heap         │
│ Stack        │     │ Stack        │
└──────────────┘     └──────────────┘
  독립된 공간            독립된 공간

프로세스 내 멀티스레드
┌─────────────────────────────────┐
│ Code    Data    Heap  (공유)     │
│ ┌──────┐ ┌──────┐ ┌──────┐     │
│ │Stack1│ │Stack2│ │Stack3│     │
│ │ T1   │ │ T2   │ │ T3   │     │
│ └──────┘ └──────┘ └──────┘     │
└─────────────────────────────────┘
```

**스레드마다 독립적으로 가지는 것**:
- **Stack**: 함수 호출 정보(매개변수, 지역변수, 복귀 주소)가 스레드마다 다르기 때문
- **레지스터(AX, BX 등)**: CPU가 현재 계산 중인 값. 스레드 A가 덧셈 도중 멈추고 스레드 B로 넘어갈 때, A의 중간 계산 값을 기억해야 하므로 독립 필수
- **PC(Program Counter)**: 각 스레드가 코드의 몇 번째 줄을 실행 중인지 (실행 위치가 스레드마다 다르므로)

**스레드끼리 공유하는 것**: Code, Data(전역 변수), Heap(동적 할당 메모리), 열린 파일(File Descriptor)

> **꼬리 질문: 스레드 간 Context Switch가 왜 빠른가요?** → 같은 프로세스 안의 스레드들은 주소 공간(페이지 테이블)이 동일하므로 **TLB flush가 불필요**하고, 공유 메모리 덕분에 **캐시 재사용률이 높습니다**. 레지스터, PC, 스택 포인터만 바꿔치기하면 되므로 훨씬 가볍습니다.

### Q. PCB(Process Control Block)에는 무엇이 저장되나요?

PCB는 프로세스에 대한 모든 정보를 담고 있는 **운영체제의 자료구조**입니다. Context Switch 시 현재 프로세스의 상태를 PCB에 저장하고, 새 프로세스의 PCB에서 상태를 복원합니다.

| 필드 | 설명 |
|------|------|
| PID | 프로세스 고유 식별자 |
| Process State | 현재 상태 (Ready, Running, Waiting 등) |
| Program Counter (PC) | 다음에 실행할 명령어 주소 |
| CPU 레지스터 | 범용 레지스터, 스택 포인터 등의 값 |
| 메모리 관리 정보 | Page Table Base Register, 세그먼트 테이블 |
| 스케줄링 정보 | 우선순위, 스케줄링 큐 포인터, 남은 time quantum |
| I/O 상태 정보 | 열린 파일 목록, I/O 장치 할당 정보 |
| Accounting 정보 | CPU 사용 시간, 시간 제한, 프로세스 번호 |

### Q. Context Switch란 무엇이고 왜 오버헤드가 발생하나요?

Context Switch는 **CPU가 현재 실행 중인 프로세스의 상태를 PCB에 저장하고, 새로 실행할 프로세스의 상태를 PCB에서 복원하는 과정**입니다.

```
프로세스 P1 실행 중
     │
     ▼ (인터럽트/시스템콜)
┌─────────────────────┐
│ P1의 상태를 PCB1에 저장  │  ← 레지스터, PC, 스택 포인터 등
│ PCB2에서 P2의 상태 복원  │  ← 레지스터, PC, 스택 포인터 등
└─────────────────────┘
     │
     ▼
프로세스 P2 실행 시작
```

**오버헤드 발생 원인**:

1. **레지스터 저장/복원**: 수십 개의 레지스터 값을 메모리에 쓰고 읽음

2. **TLB Flush**:
   - **TLB(Translation Lookaside Buffer)**는 가상 주소 → 물리 주소 변환 결과를 캐싱하는 CPU 내부의 고속 메모리
   - CPU 내부의 **MMU(Memory Management Unit)** 안에 위치하며, L1/L2/L3 데이터 캐시와는 **완전히 별개의 하드웨어**
   - 프로세스마다 페이지 테이블이 다르므로, 프로세스 전환 시 TLB의 내용이 전부 무효화됨 → **TLB Miss 급증** → 매번 느린 페이지 테이블을 참조해야 함
   - ASID(Address Space Identifier)로 TLB 엔트리에 프로세스 ID를 태깅하면 flush 없이 구분 가능 (완화 기법)
   ```
   CPU가 메모리에 접근하는 순서:
   ① TLB 확인 (가상→물리 주소 변환, MMU 안에 있음)
   ② L1/L2/L3 캐시 확인 (실제 데이터 가져오기)
   ③ 둘 다 miss면 → RAM까지 접근 (매우 느림)
   ```

3. **캐시 오염(Cache Pollution)**:
   - L1/L2/L3 캐시는 CPU가 자주 쓰는 **실제 데이터와 명령어**를 저장하는 공간
   - 프로세스 A가 실행 중일 때 캐시는 A의 데이터로 가득 참
   - 프로세스 B로 전환하면 캐시에 B의 데이터가 하나도 없음 → **Cache Miss 급증** → 느린 RAM에서 데이터를 다시 가져와야 함
   - 비유: A가 책상에 펼쳐둔 책을 전부 치우고, B가 서가에서 자기 책을 처음부터 가져오는 것

4. **파이프라인 Flush**:
   - 현대 CPU는 명령어를 여러 단계(Fetch→Decode→Execute→Writeback)로 나누어 **동시에 겹쳐서 실행**(파이프라인)
   - Context Switch가 발생하면 파이프라인에서 처리 중이던 프로세스 A의 미완성 명령어들을 **전부 폐기**해야 함
   - 새 프로세스 B의 명령어를 빈 파이프라인에 처음부터 다시 채워야 하므로 시간 소요
   - 비유: 컨베이어 벨트에서 세단을 조립하다가 갑자기 SUV로 바꾸라는 지시 → 벨트 위의 세단 부품을 다 버리고 SUV 부품을 처음부터 올려야 함

> **프로세스 간 Context Switch가 스레드 간 전환보다 비용이 큰 이유**: 스레드는 같은 주소 공간(페이지 테이블)을 공유하므로 **TLB flush가 불필요**하고, Code/Data/Heap을 공유하므로 **캐시 재사용률이 높습니다**. 스레드 전환 시에는 레지스터, PC, 스택 포인터만 바꾸면 됩니다.

### Q. 프로세스의 5가지 상태를 설명하세요.

```
         admit        dispatch
 [New] --------→ [Ready] --------→ [Running]
                    ↑                  │  │
                    │    interrupt      │  │
                    ├──────────────────┘  │
                    │                     │
                    │  I/O completion     │ I/O or event wait
                    │                     ▼
                    └──────────────── [Waiting]

                   [Running] ──exit──→ [Terminated]
```

| 상태 | 설명 | 전이 조건 |
|------|------|----------|
| **New** | 프로세스 생성됨 | → Ready: OS가 승인(admit) |
| **Ready** | CPU 할당 대기 | → Running: 스케줄러가 선택(dispatch) |
| **Running** | CPU에서 실행 중 | → Ready: time quantum 만료 또는 preemption |
| | | → Waiting: I/O 요청 또는 이벤트 대기 |
| | | → Terminated: 실행 완료(exit) |
| **Waiting** | I/O 완료 대기 | → Ready: I/O 완료 또는 이벤트 발생 |
| **Terminated** | 실행 종료 | PCB 정리 후 소멸 |

### Q. fork()와 exec()의 차이는? 왜 이 2단계 구조를 쓰나요?

**용도**: 리눅스에서 새로운 프로그램(프로세스)을 실행하기 위한 **2단 콤보**입니다.
- **fork()**: 현재 프로세스를 그대로 복제하여 분신(자식 프로세스)을 만듦
- **exec()**: 그 분신의 메모리를 새 프로그램으로 통째로 교체

```
터미널에서 "ls" 명령어를 치면 실제로 일어나는 일:

1. 셸(bash)이 fork() → 셸과 똑같은 자식 프로세스 생성
2. 자식이 exec("ls") → 자기 메모리를 ls 프로그램으로 교체
3. 부모(셸)는 waitpid()로 자식이 끝날 때까지 대기
```

| 구분 | fork() | exec() |
|------|--------|--------|
| 동작 | 현재 프로세스를 **복제**하여 자식 생성 | 현재 프로세스의 메모리를 **새 프로그램으로 교체** |
| PID | 자식은 새 PID 부여 | PID 변경 없음 |
| 메모리 | 부모의 주소 공간 복사 (COW) | 새 프로그램의 code/data/heap/stack으로 교체 |
| 반환값 | 부모: 자식 PID, 자식: 0 | 성공 시 반환하지 않음 (새 프로그램 시작) |

**fork()의 반환값이 부모/자식에게 다른 이유**:

fork()가 호출되는 순간, 프로세스가 2개로 분열됩니다. 자식은 부모의 기억을 완벽히 복사했기 때문에 **자기가 부모인지 자식인지 모릅니다**. OS가 반환값으로 이를 구분시켜 줍니다:

- **부모가 받는 값 = 자식의 PID** (예: 3045) → "네가 낳은 자식의 번호야. 이걸로 관리해"
- **자식이 받는 값 = 0** → "너는 복제본이야"

```python
pid = os.fork()  # 이 순간 프로세스가 2개로 분열!

if pid == 0:
    # [자식 프로세스]
    # "pid에 0이 들어있네? 내가 복제본이구나!"
    # "내 메모리를 ls 프로그램으로 교체하자"
    os.execvp("ls", ["ls", "-la"])
else:
    # [부모 프로세스]
    # "pid에 3045가 들어있네? 내가 본체고, 자식은 3045번이구나"
    # "자식이 끝날 때까지 기다리자"
    os.waitpid(pid, 0)
```

**COW (Copy-On-Write)**: fork() 시 실제로 메모리를 복사하지 않고, 부모와 자식이 같은 물리 페이지를 공유합니다. 어느 한쪽이 **쓰기를 시도할 때만 해당 페이지를 복사**합니다. 이로써 fork() 비용을 대폭 줄입니다.

```
fork() 직후:
  부모 가상 메모리 ──→ [물리 페이지 A] ←── 자식 가상 메모리
                       (둘 다 읽기 전용으로 공유)

자식이 쓰기 시도:
  부모 가상 메모리 ──→ [물리 페이지 A] (원본 유지)
  자식 가상 메모리 ──→ [물리 페이지 A'] (이 시점에 복사 발생!)
```

> exec() 후에는 자식의 메모리가 통째로 새 프로그램으로 교체되므로, fork() 때 COW로 복사를 미뤄두면 **실제로 복사가 한 번도 일어나지 않을 수도** 있습니다. 이것이 fork()+exec() 패턴이 효율적인 이유입니다.
