# 실습 5: 페이징 & 주소 변환

## 면접 키워드
- 페이징 (Paging)
- Page Table, Page Table Entry (PTE)
- TLB (Translation Lookaside Buffer)
- 내부 단편화 vs 외부 단편화
- 다단계 페이지 테이블
- 세그멘테이션 (Segmentation)

## 과제
페이지 테이블을 이용한 논리 주소 → 물리 주소 변환 시뮬레이터를 구현합니다.
TLB 캐시를 추가하여 주소 변환 성능 차이를 체감합니다.

## 면접 예상 질문

### Q. 페이징이란 무엇이고 왜 사용하나요?

페이징은 프로세스의 **가상 주소 공간을 고정 크기의 페이지(page)**로 나누고, 물리 메모리를 같은 크기의 **프레임(frame)**으로 나누어 **페이지를 프레임에 매핑**하는 메모리 관리 기법입니다.

```
가상 주소 공간 (프로세스)        물리 메모리
┌──────────┐               ┌──────────┐
│ Page 0   │──────────────→│ Frame 3  │ ← 비연속!
├──────────┤               ├──────────┤
│ Page 1   │──────┐        │ Frame 1  │
├──────────┤      │        ├──────────┤
│ Page 2   │──┐   └───────→│ Frame 7  │
├──────────┤  │            ├──────────┤
│ Page 3   │  └───────────→│ Frame 5  │
└──────────┘               └──────────┘
```

**장점**:
- **외부 단편화 제거**: 물리 메모리가 연속적일 필요 없음
- **메모리 보호**: 페이지별 접근 권한 설정 (Read/Write/Execute)
- **공유 용이**: 같은 프레임을 여러 프로세스가 공유 가능 (공유 라이브러리)

**주소 변환**:
```
논리 주소 (가상 주소)
┌─────────────┬───────────┐
│ 페이지 번호(p) │ 오프셋(d)  │
└──────┬──────┴───────────┘
       │ 페이지 테이블 참조
       ▼
┌─────────────┬───────────┐
│ 프레임 번호(f) │ 오프셋(d)  │  ← 오프셋은 동일!
└─────────────┴───────────┘
물리 주소
```

- **페이지 크기**: 보통 4KB (2^12)
- **페이지 번호** = 논리 주소 / 페이지 크기
- **오프셋** = 논리 주소 % 페이지 크기

### Q. Page Table의 구조와 역할은?

Page Table은 **페이지 번호 → 프레임 번호** 매핑을 저장하는 테이블입니다.

**Page Table Entry(PTE) 구성**:

| 필드 | 크기 | 설명 |
|------|------|------|
| **Frame Number** | 20bit (4KB 페이지, 32bit 시스템) | 물리 프레임 번호 |
| **Valid/Invalid bit** | 1bit | 해당 페이지가 메모리에 있는지 (0이면 page fault) |
| **Protection bit** | 3bit | Read/Write/Execute 권한 |
| **Dirty bit** | 1bit | 페이지 수정 여부 (교체 시 디스크 기록 필요) |
| **Reference bit** | 1bit | 최근 참조 여부 (페이지 교체 알고리즘에 사용) |

**Page Table의 위치**: 메모리에 저장되며, **PTBR(Page Table Base Register)**이 해당 프로세스의 페이지 테이블 시작 주소를 가리킵니다. 프로세스마다 독립된 페이지 테이블을 가집니다.

### Q. TLB란 무엇이고 왜 필요한가요?

TLB(Translation Lookaside Buffer)는 **페이지 테이블의 캐시**로, CPU 내부의 고속 연관 메모리(Associative Memory)입니다.

```
CPU가 메모리 접근 시:
  1. TLB 검색 (병렬 검색, 매우 빠름)
     ├─ TLB Hit → 프레임 번호 즉시 반환 (1 cycle)
     └─ TLB Miss → 페이지 테이블 접근 (메모리 접근 1회 추가)
  2. 프레임 번호 + 오프셋 → 물리 주소로 실제 메모리 접근
```

| 구분 | TLB 없이 | TLB 있을 때 (Hit) | TLB 있을 때 (Miss) |
|------|---------|-------------------|-------------------|
| 메모리 접근 횟수 | **2회** (테이블 + 데이터) | **1회** (데이터만) | **2회** (테이블 + 데이터) |

**Effective Access Time (EAT)** 계산:

```
EAT = hit_ratio × (TLB접근 + 메모리접근) + (1-hit_ratio) × (TLB접근 + 2×메모리접근)

예: TLB접근=10ns, 메모리접근=100ns, hit_ratio=98%
EAT = 0.98×(10+100) + 0.02×(10+200) = 107.8 + 4.2 = 112ns
TLB 없으면: 200ns → 약 44% 성능 향상
```

> Context Switch 시 TLB를 flush해야 합니다 (프로세스마다 페이지 테이블이 다르므로). **ASID(Address Space Identifier)**를 사용하면 TLB 엔트리에 프로세스 ID를 태깅하여 flush 없이 구분 가능합니다.

### Q. 내부 단편화와 외부 단편화의 차이는?

| 구분 | 내부 단편화 (Internal) | 외부 단편화 (External) |
|------|---------------------|---------------------|
| 정의 | 할당된 공간 중 **사용하지 않는** 부분 | 총 빈 공간은 충분하지만 **연속되지 않아** 할당 불가 |
| 발생 원인 | 고정 크기 할당 (페이징) | 가변 크기 할당 (세그멘테이션) |
| 예시 | 4KB 페이지에 3.5KB만 사용 → 0.5KB 낭비 | 10KB 빈 공간이 3곳에 분산 → 8KB 할당 불가 |
| 해결 | 페이지 크기 줄이기 (but 테이블 커짐) | 페이징, Compaction |

```
내부 단편화 (페이징):
  프로세스 크기: 13KB, 페이지 크기: 4KB
  → 4개 페이지 필요 (4+4+4+1KB) → 마지막 페이지 3KB 낭비

외부 단편화 (연속 할당):
  메모리: [사용|3KB빈|사용|5KB빈|사용|2KB빈] = 총 10KB 빈
  → 8KB 프로세스 할당 불가 (연속 8KB 없음)
```

### Q. 다단계 페이지 테이블은 왜 사용하나요?

32bit 주소 공간에서 4KB 페이지를 사용하면:
- 페이지 수 = 2^32 / 2^12 = **2^20 = 약 100만 개**
- PTE 4바이트 × 100만 = **4MB의 연속 메모리 필요** (프로세스마다!)

대부분의 프로세스는 주소 공간의 극히 일부만 사용하므로 **대부분의 PTE가 낭비**됩니다.

**2단계 페이지 테이블**:

```
논리 주소 (32bit, 4KB 페이지):
┌────────┬────────┬──────────┐
│ p1(10) │ p2(10) │ d (12)   │
└───┬────┴───┬────┴──────────┘
    │        │
    ▼        ▼
 Outer     Inner
 Table     Table → 프레임 번호

- Outer Table: 1024개 엔트리 (2^10) → 4KB
- Inner Table: 각 1024개 엔트리 → 4KB (사용하는 영역만 생성!)
```

**장점**: 사용하지 않는 영역의 Inner Table은 생성하지 않아 **메모리 절약**

**64bit 시스템 (x86-64)**: 4단계 페이지 테이블 사용 (PML4 → PDPT → PD → PT)
- 실제로는 48bit만 사용 (256TB 주소 공간)
