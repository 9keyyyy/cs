# 실습 3: DNS Resolver 구현

## 면접 키워드
- DNS 질의 과정 (재귀 질의 vs 반복 질의)
- DNS 레코드 타입 (A, AAAA, CNAME, MX, NS)
- TTL과 캐싱
- DNS 계층 구조 (Root → TLD → Authoritative)

## 과제

### Part A: DNS 질의 직접 보내기
UDP 소켓으로 DNS 서버(8.8.8.8)에 직접 질의를 보내고 응답을 파싱합니다.

### Part B: 로컬 DNS 캐시 구현
TTL 기반 캐싱을 구현하여 동일 도메인 재질의 시 캐시에서 응답합니다.

## 면접 예상 질문

### Q. 브라우저에 www.google.com을 입력하면 DNS 질의 과정은?

1. **브라우저 캐시 확인**: 브라우저가 자체 DNS 캐시에서 해당 도메인의 IP를 찾습니다.
2. **OS 캐시 확인**: 브라우저 캐시에 없으면 운영체제의 DNS 캐시(hosts 파일 포함)를 확인합니다.
3. **로컬 DNS 서버(리졸버) 질의**: OS 캐시에도 없으면 ISP 또는 설정된 DNS 서버(예: 8.8.8.8)에 **재귀 질의(Recursive Query)** 를 보냅니다. 클라이언트는 최종 답을 요구하고, 나머지 탐색은 리졸버가 대행합니다.
4. **리졸버의 반복 질의(Iterative Query)**: 리졸버는 계층적으로 질의합니다.
   - **루트 DNS 서버** → `.com` TLD 서버 주소를 응답
   - **TLD(.com) DNS 서버** → `google.com`의 Authoritative 서버 주소를 응답
   - **Authoritative DNS 서버** → `www.google.com`의 실제 IP 주소를 응답
5. **응답 반환 및 캐싱**: 리졸버가 받은 IP를 클라이언트에 전달하고, 각 단계의 결과를 TTL에 따라 캐싱합니다.

> 재귀 질의는 클라이언트→리졸버 구간에서, 반복 질의는 리졸버→각 네임서버 구간에서 사용됩니다.

### Q. DNS가 UDP를 사용하는 이유는?

DNS 질의는 일반적으로 **하나의 요청과 하나의 응답**으로 완료되는 단순한 구조입니다. UDP를 사용하는 이유는 다음과 같습니다.

1. **빠른 응답**: TCP의 3-way handshake 없이 즉시 질의를 보낼 수 있어 지연 시간이 짧습니다.
2. **작은 메시지 크기**: 대부분의 DNS 메시지는 512바이트 이내로, UDP 데이터그램 하나에 충분히 담깁니다.
3. **연결 상태 불필요**: DNS 서버는 초당 수많은 질의를 처리해야 하므로, 각 요청마다 TCP 연결 상태를 유지하면 리소스 낭비가 큽니다.
4. **신뢰성은 애플리케이션 계층에서 처리**: 응답이 오지 않으면 클라이언트가 단순히 재질의하면 되므로 TCP 수준의 신뢰성이 필요하지 않습니다.

**예외적으로 TCP를 사용하는 경우**:
- 응답 메시지가 512바이트를 초과할 때 (TC 플래그 설정 후 TCP로 재질의)
- DNS 영역 전송(Zone Transfer, AXFR) 시 — 대량의 레코드를 안정적으로 전송해야 하므로

### Q. DNS 캐시 포이즈닝이란?

DNS 캐시 포이즈닝(Cache Poisoning)은 **DNS 리졸버의 캐시에 위조된 레코드를 삽입하는 공격**입니다.

**공격 원리**:
1. 공격자가 리졸버에게 특정 도메인 질의를 유도합니다.
2. 리졸버가 Authoritative 서버에 반복 질의를 보내고 응답을 기다리는 사이, 공격자가 위조된 응답을 먼저 보냅니다.
3. 위조 응답의 Transaction ID가 일치하면 리졸버가 이를 정상 응답으로 받아들여 캐시에 저장합니다.
4. 이후 해당 도메인을 질의하는 모든 사용자가 공격자의 서버(피싱 사이트 등)로 유도됩니다.

**방어 방법**:
- **DNSSEC**: DNS 응답에 전자 서명을 추가하여 응답의 진위를 검증
- **Source Port Randomization**: 질의의 출발 포트를 랜덤화하여 위조 응답이 일치할 확률을 낮춤
- **Transaction ID 랜덤화**: 16비트 ID를 예측하기 어렵게 만듦

### Q. TTL의 역할은?

TTL(Time To Live)은 **DNS 레코드가 캐시에 유지되는 시간(초)**을 의미합니다. Authoritative DNS 서버가 응답에 TTL 값을 포함하여 전달합니다.

**역할**:
- **캐시 유효 기간 설정**: TTL이 만료되면 캐시에서 해당 레코드를 삭제하고 다시 질의합니다.
- **부하 분산**: TTL이 길면 캐시 히트율이 높아져 DNS 서버 부하가 줄어듭니다.
- **변경 반영 속도 제어**: TTL이 짧으면 DNS 레코드 변경 시 빠르게 전파됩니다.

**TTL 설정 전략**:
| TTL | 용도 |
|-----|------|
| 짧음 (60~300초) | IP가 자주 변경되는 서비스, 장애 대응 시 빠른 전환 필요 |
| 김 (3600~86400초) | IP가 거의 변경되지 않는 안정적인 서비스, DNS 서버 부하 감소 |

> DNS 마이그레이션 시에는 사전에 TTL을 낮춰두고, 전환 완료 후 다시 높이는 것이 일반적인 전략입니다.
