## 면접 예상 질문

### Q. TCP와 UDP의 차이를 설명하세요.

TCP는 연결 지향(Connection-oriented) 프로토콜로, 데이터를 전송하기 전에 3-way handshake를 통해 연결을 수립합니다. 데이터의 순서 보장, 재전송을 통한 신뢰성, 흐름 제어와 혼잡 제어를 제공합니다. 반면 UDP는 비연결(Connectionless) 프로토콜로, 연결 수립 과정 없이 즉시 데이터를 전송합니다. 순서 보장이나 재전송을 하지 않아 신뢰성은 낮지만, 그만큼 오버헤드가 적어 지연 시간이 짧습니다.

| 구분 | TCP | UDP |
|------|-----|-----|
| 연결 방식 | 연결 지향 (3-way handshake) | 비연결 |
| 신뢰성 | 보장 (재전송, ACK) | 미보장 |
| 순서 보장 | O (시퀀스 번호) | X |
| 흐름/혼잡 제어 | O | X |
| 헤더 크기 | 20~60바이트 | 8바이트 |
| 속도 | 상대적으로 느림 | 빠름 |

### Q. 3-way handshake 과정을 설명하세요.

TCP 연결 수립 시 클라이언트와 서버 간에 3단계로 이루어지는 과정입니다.

1. **SYN**: 클라이언트가 서버에게 연결 요청을 보냅니다. 이때 클라이언트는 자신의 초기 시퀀스 번호(ISN)를 함께 전달합니다.
2. **SYN-ACK**: 서버가 클라이언트의 요청을 수락하며, 서버 자신의 초기 시퀀스 번호와 함께 클라이언트의 SYN에 대한 ACK를 응답합니다.
3. **ACK**: 클라이언트가 서버의 SYN-ACK에 대한 ACK를 보내면 연결이 수립됩니다.

이 과정을 통해 양쪽 모두 데이터를 송수신할 준비가 되었음을 확인하고, 초기 시퀀스 번호를 교환하여 이후 데이터 전송의 순서를 추적할 수 있게 됩니다.

> 연결 해제 시에는 4-way handshake(FIN → ACK → FIN → ACK)를 사용합니다.

### Q. UDP를 사용하는 서비스의 예시와 그 이유는?

- **DNS**: 짧은 질의/응답 구조로, 연결 수립 오버헤드 없이 빠르게 처리하는 것이 유리합니다.
- **실시간 스트리밍(영상/음성)**: 약간의 패킷 손실보다 지연 시간이 더 치명적이므로, 재전송 없이 빠르게 전달하는 UDP가 적합합니다.
- **온라인 게임**: 캐릭터 위치 등 실시간 데이터는 과거 데이터를 재전송받기보다 최신 데이터를 빠르게 받는 것이 중요합니다.
- **DHCP**: 클라이언트가 아직 IP를 할당받기 전이므로 TCP 연결 수립이 불가능하여 UDP 브로드캐스트를 사용합니다.

공통적으로 **낮은 지연이 신뢰성보다 중요하거나**, **연결 수립 자체가 불가능/비효율적인 상황**에서 UDP를 선택합니다.

### Q. TCP가 신뢰성을 보장하는 메커니즘은?

1. **시퀀스 번호와 ACK**: 각 바이트에 시퀀스 번호를 부여하고, 수신 측이 ACK로 확인 응답을 보내 데이터 수신을 확인합니다.
2. **재전송(Retransmission)**: 일정 시간(RTO) 내에 ACK를 받지 못하면 데이터를 재전송합니다.
3. **체크섬(Checksum)**: 세그먼트의 무결성을 검증하여 손상된 데이터를 감지합니다.
4. **흐름 제어(Flow Control)**: 슬라이딩 윈도우 방식으로 수신 측의 처리 능력에 맞춰 전송 속도를 조절합니다.
5. **혼잡 제어(Congestion Control)**: 네트워크 혼잡 상태를 감지하고 전송 속도를 조절하여 패킷 손실을 줄입니다(Slow Start, Congestion Avoidance 등).
6. **순서 보장**: 시퀀스 번호를 기반으로 수신 측에서 데이터를 올바른 순서로 재조립합니다.
