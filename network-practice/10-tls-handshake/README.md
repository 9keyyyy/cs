## 면접 예상 질문

### Q. HTTPS의 동작 원리를 설명하세요.

HTTPS는 **HTTP + TLS**로, 기존 HTTP 통신을 TLS(Transport Layer Security) 프로토콜로 감싼 것입니다.

```
[클라이언트] ←──── HTTP (평문) ────→ [서버]         ← HTTP
[클라이언트] ←─ TLS(HTTP 암호화) ──→ [서버]         ← HTTPS
```

**동작 순서**:
1. **TCP 3-Way Handshake** — 일반적인 TCP 연결 수립
2. **TLS Handshake** — 암호화 알고리즘 협상, 서버 인증, 세션 키 생성
3. **암호화 통신** — 생성된 대칭 세션 키로 HTTP 데이터를 암호화하여 송수신
4. **연결 종료** — TLS 세션 종료 후 TCP 연결 해제

핵심은 **비대칭키로 안전하게 대칭키를 교환**한 뒤, 실제 데이터는 **대칭키로 암호화**하여 성능과 보안을 모두 확보하는 것입니다.

### Q. 대칭키와 비대칭키의 차이는?

| 구분 | 대칭키 (Symmetric) | 비대칭키 (Asymmetric) |
|------|-------------------|----------------------|
| **키 구조** | 암호화/복호화에 **같은 키** 사용 | **공개키**(암호화) + **개인키**(복호화) 쌍 |
| **속도** | 빠름 (AES 등) | 느림 (RSA, ECDHE 등) |
| **키 교환 문제** | 키를 안전하게 전달하기 어려움 | 공개키를 공개해도 안전 |
| **용도** | 데이터 암호화 (본 통신) | 키 교환, 디지털 서명, 인증 |
| **대표 알고리즘** | AES, ChaCha20 | RSA, ECDHE, Ed25519 |

```
대칭키:
  A ──[key]──암호화──→ 암호문 ──[같은 key]──복호화──→ B

비대칭키:
  A ──[B의 공개키]──암호화──→ 암호문 ──[B의 개인키]──복호화──→ B
```

TLS에서는 **비대칭키로 대칭 세션 키를 안전하게 교환**하고, 이후 **대칭키로 실제 데이터를 암호화**합니다. 두 방식의 장점을 결합한 **하이브리드 암호화** 방식입니다.

### Q. TLS Handshake 과정을 단계별로 설명하세요.

**TLS 1.2 기준** (2-RTT):

```
Client                                           Server
  │                                                 │
  │──── 1. ClientHello ────────────────────────────→│
  │     (TLS 버전, 지원 암호 목록, Client Random)      │
  │                                                 │
  │←─── 2. ServerHello ────────────────────────────│
  │     (선택된 암호, Server Random)                   │
  │←─── 3. Certificate ───────────────────────────│
  │     (서버 인증서 + 공개키)                          │
  │←─── 4. ServerKeyExchange ─────────────────────│
  │     (ECDHE 파라미터, 서명)                         │
  │←─── 5. ServerHelloDone ───────────────────────│
  │                                                 │
  │──── 6. ClientKeyExchange ─────────────────────→│
  │     (클라이언트 ECDHE 공개값)                       │
  │──── 7. ChangeCipherSpec ──────────────────────→│
  │──── 8. Finished (암호화) ─────────────────────→│
  │                                                 │
  │←─── 9. ChangeCipherSpec ──────────────────────│
  │←── 10. Finished (암호화) ─────────────────────│
  │                                                 │
  │←════ 암호화된 애플리케이션 데이터 통신 ════════→│
```

**핵심 단계 요약**:
1. **협상** — 클라이언트와 서버가 사용할 암호 스위트, TLS 버전 합의
2. **인증** — 서버가 인증서를 제공하고, 클라이언트가 CA 체인으로 검증
3. **키 교환** — ECDHE 등으로 양측이 독립적으로 **Pre-Master Secret** 생성
4. **세션 키 도출** — Client Random + Server Random + Pre-Master Secret → **Master Secret** → 대칭 세션 키
5. **암호화 전환** — 이후 모든 통신은 대칭키로 암호화

**TLS 1.3**에서는 이 과정이 **1-RTT**로 단축됩니다 (ClientHello에 키 교환 파라미터를 포함).

### Q. 인증서의 역할은? CA란?

**인증서(Certificate)**는 **서버의 공개키가 진짜 해당 서버의 것임을 보증**하는 디지털 문서입니다.

```
인증서 내용:
┌──────────────────────────────────┐
│ Subject: www.example.com         │  ← 이 도메인의 인증서
│ Public Key: 서버의 공개키          │  ← 클라이언트가 사용할 공개키
│ Issuer: Let's Encrypt CA         │  ← 발급한 CA
│ Valid: 2025-01-01 ~ 2026-01-01   │  ← 유효기간
│ Signature: CA의 개인키로 서명       │  ← 위조 방지
└──────────────────────────────────┘
```

인증서가 없다면 클라이언트는 받은 공개키가 **서버의 것인지 공격자의 것인지 구분할 수 없습니다** (중간자 공격, MITM).

**CA(Certificate Authority)**는 인증서를 **발급하고 서명하는 신뢰된 제3자 기관**입니다.

```
신뢰 체인 (Chain of Trust):
Root CA (OS/브라우저에 내장)
  └── Intermediate CA (Root CA가 서명)
        └── 서버 인증서 (Intermediate CA가 서명)
```

**검증 과정**:
1. 서버가 인증서를 보냄
2. 클라이언트가 **Issuer(발급 CA)의 공개키**로 서명 검증
3. 해당 CA의 인증서를 또 상위 CA로 검증 → **Root CA**까지 체인을 타고 올라감
4. Root CA는 OS/브라우저에 **사전 내장**되어 있으므로 신뢰

### Q. HTTP와 HTTPS의 차이는?

| 구분 | HTTP | HTTPS |
|------|------|-------|
| **정식 명칭** | HyperText Transfer Protocol | HyperText Transfer Protocol **Secure** |
| **포트** | 80 | 443 |
| **암호화** | 없음 (평문 전송) | TLS로 암호화 |
| **인증** | 서버 신원 확인 불가 | 인증서로 서버 신원 확인 |
| **무결성** | 데이터 변조 감지 불가 | MAC/AEAD로 변조 감지 |
| **속도** | 약간 빠름 | TLS Handshake 오버헤드 (초기 연결 시) |

**보안 위협 비교**:
```
HTTP (평문):
  클라이언트 ──"password=1234"──→ [공격자가 도청 가능] ──→ 서버
                                  [공격자가 변조 가능]

HTTPS (암호화):
  클라이언트 ──"aX9#k2$m..."───→ [공격자: 해독 불가] ───→ 서버
```

HTTP는 **기밀성, 무결성, 인증** 세 가지가 모두 없으므로 도청(Sniffing), 변조(Tampering), 위장(Spoofing) 공격에 취약합니다. HTTPS는 TLS를 통해 이 세 가지를 모두 보장합니다.

현재는 브라우저가 HTTP 사이트에 **"주의 요함"** 경고를 표시하며, SEO에서도 HTTPS가 우대받아 **사실상 HTTPS가 표준**입니다.
